# Extract Table

Extract labeled tables from one PDF. The main deliverable is structured data, not just an image crop.

Rendered page images are ground truth. PDF text/table helpers are candidate evidence only; do not accept their rows, columns, merged headers, or footnotes without visual comparison against the rendered page.

Raw table extraction often flattens multi-level headers, splits wrapped cells into false rows, and merges footnotes into body rows when they share the same column grid. Decide table structure from visual evidence, not parser output.

## Modes

- `initial_extraction`: scan the assigned scope and produce the full table artifact contract.
- `initial_extraction_batch`: same work, but write only inside the batch output directory assigned by the orchestrator.
- `continue_incomplete_extraction`: audit existing artifacts first, then continue from the first missing contract item.
- `repair_from_visual_review`: read `table_repair_requests.json` and repair only requested table crops/structure/cell text unless a linked neighboring table must change.

In repair mode, start by reading the repair request and the current table artifacts; do not redo whole-paper discovery unless the request says discovery is wrong.

## File Contract

Read:

- `<pdf_path>`
- `<output_dir>/pages/page_<N>.png`, creating pages with `render_pages.py` if needed
- optional raw candidates from `skills/extract-table/scripts/extract_tables_raw.py`
- in repair mode: `<output_dir>/tables/table_repair_requests.json`

Write under `<output_dir>/tables/`:

- `table_candidates.json`
- `table_index.json`
- `table_decisions.json`
- `Table_<N>.png`
- `Table_<N>.json`
- bounded previews under `previews/`
- `tables.json`
- `tables_mechanical_validation.json`

If no labeled tables exist, write empty `table_candidates.json`, `table_index.json`, `table_decisions.json`, and `tables.json`, then run mechanical validation.

In `initial_extraction_batch` mode, write all the same artifacts under the batch output directory, including a batch-local `table_candidates.json` covering only the assigned pages. The merge script requires all four manifest files per batch: `table_candidates.json`, `table_index.json`, `table_decisions.json`, `tables.json`.

Before writing any JSON manifest, read `../_shared/references/table_schemas.md` for exact field names and shapes.

## Image Rule

Do not read full-resolution rendered pages, full-resolution crops, or raw high-resolution strips directly. Read bounded previews only: both dimensions <=1600 px, and <=1400 px for multi-image reads.

Use:

```bash
python3 ${CLAUDE_SKILL_DIR}/../_shared/scripts/crop_region.py ...
python3 ${CLAUDE_SKILL_DIR}/../_shared/scripts/make_image_preview.py ...
```

## Scope Rules

Include labeled table content: table label, title, headers (including spanning/merged headers as additional levels in `headers`), body rows, units, footnote markers, and table footnotes.

Exclude page headers/footers, body prose, figure captions, unrelated equations, adjacent tables, and page chrome (journal running headers, author name lines, page numbers, separator rules, publisher logos, DOI footers).

Tables embedded as figure panels are not extracted here unless the user explicitly asks for figure-internal tables.

## Workflow

1. Render pages if needed.
2. Optionally run raw candidate extraction:
   ```bash
   python3 ${CLAUDE_SKILL_DIR}/scripts/extract_tables_raw.py "<pdf_path>" "<output_dir>/tables/raw"
   ```
3. Read bounded page previews for the assigned scope. Treat page hints from the orchestrator as hints, not truth.
4. Write `table_candidates.json` with all labeled tables found, unexpected labeled tables, and assigned pages scanned.
5. Write `table_index.json` to commit which candidates are real labeled tables, using ID, label/title, pages, candidate IDs, and notes only. Structure counts belong in `table_decisions.json`.
6. For each indexed table, read a bounded source region that shows the title, full table body, footnote boundary, and nearby non-table content.
7. Decide crop boundaries and table structure visually. Before committing coordinates, create and inspect edge previews:
   - **Top**: no journal header, running title, page number, or separator rule above the table label.
   - **Bottom**: no page number, body prose, publisher footer, or logo below the last footnote.
   - **Left/Right**: no margin annotations, adjacent-table content, or column bleed.
   Shrink inward to the nearest clean gap if page chrome is within the proposed boundary.
8. Write `table_decisions.json` before final serialization.
9. Crop `Table_<N>.png`, create crop previews, and serialize `Table_<N>.json`. CSV is generated by the orchestrator before final validation.
10. Render each structured table for visual comparison:
   ```bash
   python3 ${CLAUDE_SKILL_DIR}/../_shared/scripts/render_table_preview.py \
     "<output_dir>/tables/Table_1.json" "<output_dir>/tables/previews/Table_1_rendered.png"
   ```
11. Read the source region, crop preview, and rendered table preview. Fix defects before writing a pass.
12. Write `tables.json`.
13. Run:
   ```bash
   python3 ${CLAUDE_SKILL_DIR}/../_shared/scripts/validate_tables.py \
     "<output_dir>" --mechanical-only --write "<output_dir>/tables/tables_mechanical_validation.json"
   ```

If validation fails, repair the artifacts and rerun it before returning.

## Structured Data Rules

- `headers` is always an array of string arrays. Single-level: `[["A", "B", "C"]]`; multi-row or spanning headers: `[["", "Group", ""], ["A", "B", "C"]]`.
- Every data row must have the same width as the lowest header level.
- Empty cells are `""`, never `null`.
- Replace wrapped line breaks inside cells with spaces.
- Preserve author text exactly as printed in the source, including misspellings, broken words, and unusual formatting. Do not silently correct grammar or spelling. The only allowed transformations: normalizing line breaks to spaces within cells, and standardizing Unicode representations of the same character.
- Title separator fidelity: reproduce the exact separator between table number and descriptive text (period, em-dash, colon, etc.) — do not normalize.
- Footnote marker prefix: when a source footnote begins with a marker (ᵃ, *, †, superscript number), the `footnotes` array entry must begin with that same marker.
- If a cell cannot be confidently read, record the uncertainty in `notes`; do not silently guess and pass.
- When the first column has visually distinct sub-columns, split them into separate data columns. If the source header spans both sub-columns as a single label, add a level to `headers` with the source label in the first position and `""` for the unlabeled sub-column. Do not invent header names that don't appear in the source.
- Multi-page tables: one crop per page, named `Table_<N>_p<page>.png` (e.g., `Table_3_p10.png`, `Table_3_p11.png`). List all page crops in `image_files`. A combined stitched `Table_<N>.png` is optional.

## Verification

A table is mechanically complete only when the crop and structured output both match the rendered page evidence:

- table label/title complete
- all header levels represented
- row and column counts match the source
- wrapped cells are not split into false rows
- merged/spanning headers are not flattened incorrectly
- footnotes and markers are present
- no surrounding prose/page chrome leaks into the crop
- top, bottom, left, and right crop edges are clean
- sub-column structure verified (if source has visual sub-columns)

Use only `pass` or `fail` in verification fields. Uncertainty is `fail` until repaired or recorded as a terminal condition by the orchestrator.

## Repair Mode

Repair requests describe the defect and direction, not exact coordinates. Valid repair directions include `expand_crop_bottom`, `shrink_crop_top`, `split_header_level`, `merge_wrapped_cell`, `split_false_row`, `correct_cell_text`, and `add_missing_footnote`.

After repair, update `table_decisions.json`, affected `Table_<N>.json`/crop/previews, `tables.json`, and `tables_mechanical_validation.json`. When repairing cell content, headers, title, or footnotes, update both `Table_<N>.json` and the matching structured fields in `table_decisions.json`; the validator compares them. Do not edit old round snapshots.

# 擷取表格

## 中繼資料

- name: `extract-table`
- description: 將有標籤的科學表格擷取成已驗證的結構化 JSON 和來源裁切圖；已轉成圖片的 PDF 頁面是最終依據，原始表格工具只能作為候選證據。CSV 匯出由協調器產生。
- argument-hint: `<pdf_path> <output_dir> [mode]`

## 任務

從一份 PDF 擷取有標籤的表格。主要交付成果是結構化資料，而不只是圖片裁切。

已轉成圖片的頁面是最終依據。PDF 文字或表格輔助工具只提供候選證據；在沒有和頁面圖片做視覺比較前，不要接受它們的列、欄、合併標題或註腳。

原始表格擷取常會把多層標題壓平，把換行儲存格拆成假列，並在註腳和正文列共用同一欄格時，把註腳合併進正文列。表格結構要根據視覺證據決定，不要根據解析器輸出決定。

## 模式

- `initial_extraction`：掃描指派範圍，並產生完整表格成果契約。
- `initial_extraction_batch`：工作相同，但只寫入協調器指派的批次輸出目錄。
- `continue_incomplete_extraction`：先稽核既有成果，再從第一個缺失的契約項目繼續。
- `repair_from_visual_review`：讀取 `table_repair_requests.json`，只修復被要求的表格裁切、結構或儲存格文字；除非相鄰表格也必須一起變更。

在修復模式中，要先讀修復請求和目前表格成果；除非請求指出發現步驟本身錯誤，不要重新做整篇論文搜尋。

## 檔案契約

讀取：

- `<pdf_path>`
- `<output_dir>/pages/page_<N>.png`，必要時用 `render_pages.py` 建立。
- `skills/extract-table/scripts/extract_tables_raw.py` 的選用原始候選輸出。
- 修復模式中的 `<output_dir>/tables/table_repair_requests.json`。

寫到 `<output_dir>/tables/`：

- `table_candidates.json`
- `table_index.json`
- `table_decisions.json`
- `Table_<N>.png`
- `Table_<N>.json`
- `previews/` 下的受限尺寸預覽圖
- `tables.json`
- `tables_mechanical_validation.json`

如果沒有有標籤的表格，仍要寫出空的 `table_candidates.json`、`table_index.json`、`table_decisions.json` 和 `tables.json`，然後執行機械式驗證。

在 `initial_extraction_batch` 模式中，也要在批次輸出目錄下寫出相同成果，包括只涵蓋指派頁面的批次本地 `table_candidates.json`。合併腳本要求每個批次都有四個清單檔：`table_candidates.json`、`table_index.json`、`table_decisions.json`、`tables.json`。

寫任何 JSON 清單檔前，先讀 `../_shared/references/table_schemas.md`，取得精確欄位名稱和形狀。

## 圖片規則

不要直接讀完整解析度的頁面圖片、完整解析度的裁切圖，或原始高解析度條帶。只讀受限尺寸預覽圖：單張圖片兩邊都不超過 1600 px；多圖讀取時兩邊都不超過 1400 px。

使用：

```bash
python3 ${CLAUDE_SKILL_DIR}/../_shared/scripts/crop_region.py ...
python3 ${CLAUDE_SKILL_DIR}/../_shared/scripts/make_image_preview.py ...
```

## 範圍規則

包含有標籤的表格內容：表格標籤、標題、欄標題、資料列、跨欄或合併標題（在 `headers` 中以額外層級表示）、單位、註腳標記和表格註腳。

排除頁首/頁尾、正文、圖說、無關方程式、相鄰表格和頁面固定元素，例如期刊頁首、作者列、頁碼、分隔線、出版商 logo、DOI 頁尾。

嵌在圖面板中的表格不在此擷取，除非使用者明確要求擷取圖內表格。

## 工作流程

1. 必要時轉出頁面圖片。
2. 可選擇執行原始候選擷取：
   ```bash
   python3 ${CLAUDE_SKILL_DIR}/scripts/extract_tables_raw.py "<pdf_path>" "<output_dir>/tables/raw"
   ```
3. 讀取指派範圍的受限尺寸頁面預覽圖。協調器提供的頁面提示只是提示，不是最終依據。
4. 寫 `table_candidates.json`，包含所有找到的有標籤表格、意外發現的有標籤表格，以及已掃描頁面。
5. 寫 `table_index.json`，決定哪些候選項是真正的有標籤表格。此檔只放 ID、標籤/標題、頁面、候選 ID 和註記；結構數量放在 `table_decisions.json`。
6. 對每個已列入索引的表格，讀取受限尺寸來源區域，必須顯示標題、完整表身、註腳邊界和附近非表格內容。
7. 根據視覺證據決定裁切邊界和表格結構。提交座標前，建立並檢查邊緣預覽圖：
   - **上邊**：表格標籤上方不能包含期刊頁首、running title、頁碼或分隔線。
   - **下邊**：最後一個註腳下方不能包含頁碼、正文、出版商頁尾或 logo。
   - **左/右邊**：不能包含頁邊註記、相鄰表格內容或欄位外溢。
   如果頁面固定元素在擬定邊界內，向內縮到最近的乾淨間隔。
8. 在最終序列化前寫 `table_decisions.json`。
9. 裁切 `Table_<N>.png`，建立裁切預覽圖，並序列化 `Table_<N>.json`。CSV 由協調器在最終驗證前產生。
10. 將每個結構化表格轉成視覺預覽：
   ```bash
   python3 ${CLAUDE_SKILL_DIR}/../_shared/scripts/render_table_preview.py \
     "<output_dir>/tables/Table_1.json" "<output_dir>/tables/previews/Table_1_rendered.png"
   ```
11. 讀取來源區域、裁切預覽圖和轉出的表格預覽圖。通過前修正缺陷。
12. 寫 `tables.json`。
13. 執行：
   ```bash
   python3 ${CLAUDE_SKILL_DIR}/../_shared/scripts/validate_tables.py \
     "<output_dir>" --mechanical-only --write "<output_dir>/tables/tables_mechanical_validation.json"
   ```

如果驗證未通過，先修復成果並重新執行驗證，再回覆。

## 結構化資料規則

- `headers` 永遠是 array of string arrays。單層：`[["A", "B", "C"]]`；多列或跨欄標題：`[["", "Group", ""], ["A", "B", "C"]]`。
- 每個資料列的寬度都必須等於最低層標題列的寬度。
- 空儲存格使用 `""`，絕不使用 `null`。
- 儲存格內換行要改成空格。
- 完整保留作者在來源中印出的文字，包括錯字、斷字和不尋常格式。不要靜默修正文法或拼字。唯一允許的轉換是把儲存格內換行正規化成空格，以及把同一字元的 Unicode 表示標準化。
- 標題分隔符要忠實：表號和描述文字之間的句點、破折號、冒號等要照原文，不要正規化。
- 註腳若以標記開頭，例如 ᵃ、*、† 或上標數字，`footnotes` 陣列項目也必須以同一標記開頭。
- 如果某個儲存格無法有把握讀取，在 `notes` 記錄不確定性；不要默默猜測後判定通過。
- 如果第一欄在視覺上有不同子欄，將它們拆成獨立資料欄。若來源標題以單一標籤橫跨兩個子欄，在 `headers` 中增加一層，第一個位置放來源標籤，未標示的子欄放 `""`。不要發明來源沒有的標題名稱。
- 跨頁表格每頁一個裁切圖，命名為 `Table_<N>_p<page>.png`，例如 `Table_3_p10.png`、`Table_3_p11.png`。在 `image_files` 中列出所有頁面裁切圖。合併拼接的 `Table_<N>.png` 是選用項。

## 驗證

只有當裁切圖和結構化輸出都符合頁面圖片證據時，表格才算機械式完成：

- 表格標籤和標題完整。
- 所有標題層級都已表示。
- 列數和欄數符合來源。
- 換行儲存格沒有被拆成假列。
- 合併或跨欄標題沒有被錯誤壓平。
- 註腳和註腳標記存在。
- 裁切圖沒有包含周邊正文或頁面固定元素。
- 上、下、左、右裁切邊緣乾淨。
- 如果來源有視覺子欄，已驗證子欄結構。

驗證欄位只用 `pass` 或 `fail`。只要仍有不確定性，就使用 `fail`，直到修復完成，或由協調器記錄為終止狀態。

## 修復模式

修復請求描述缺陷和方向，不提供精確座標。有效修復方向包括 `expand_crop_bottom`、`shrink_crop_top`、`split_header_level`、`merge_wrapped_cell`、`split_false_row`、`correct_cell_text` 和 `add_missing_footnote`。

修復後，更新 `table_decisions.json`、受影響的 `Table_<N>.json`/裁切圖/預覽圖、`tables.json` 和 `tables_mechanical_validation.json`。

修復儲存格內容、標題、表題或註腳時，要同時更新 `Table_<N>.json` 和 `table_decisions.json` 中對應的結構化欄位；驗證器會比較兩者。不要編輯舊輪次快照。
