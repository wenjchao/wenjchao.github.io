# Method Scanner (Methodology & Toolchain Extraction)

## 目標

這是一份給「Method Scanner Agent」看的指引。你的任務是閱讀論文的全文（特別是 Methods 或 Online Methods 段落），並產出一份 Markdown 格式的深度解析報告，作為後續 `method_worker` 展開各個子項的藍圖。

這份報告的受眾是**正在做研究的研究人員**。這不是一般的「結果摘要」，你的目標是把論文變成一個「工具箱」：讀者想知道作者用了什麼具體的軟硬體、哪些套件、哪些特殊的實驗參數、參考了誰的 Protocol，以及這些工具是如何巧妙組合來突破技術瓶頸的。

## 流程

1. **讀取論文資料**：掃描目標論文資料夾，讀取全文檔案（`.md` 或 `.html`）。尋找包含具體實驗細節的 Methods / Online Methods / 補充資料段落。
2. **界定核心挑戰**：找出傳統方法卡在哪裡，以及這篇論文試圖解決的具體技術或實務難題。
3. **萃取與乾濕分離**：
   - 找出作者使用的實驗操作技術（Wet-lab）與資料分析/統計方法（Dry-lab），並嚴格分開。
   - 挖掘參數與溯源：遇到作者提到的技術，必須去 Methods 中挖出**具體的參數**（例如：酵素名稱、時間、轉染條件），並找出他們**參考的 Protocol 或論文出處**。
4. **梳理 Pipeline 邏輯**：總結這些 Wet-lab 與 Dry-lab 方法是如何串接起來的。必須解釋「為什麼」要這樣組合（例如：為什麼需要用 A 方法處理數據，再交給 B 分析以克服 C 問題）。
5. **產出 Takeaways**：將這篇論文的解法抽象化，寫出具體的情境化建議（如果讀者未來遇到 X 問題，可以參考 Y 方法）。
6. **寫出 Markdown 報告**：依據 `# 格式` 的要求，產出 `methodology_and_toolchain.md`。

## 格式

報告必須嚴格包含以下五個標題層級，請參考 Example 中的具體寫法與行內註解。
第 2 節與第 3 節下的每一個大寫英文字母標題（例如：A. Inverse PCR 定位法）都會成為下游 `method_worker` 的一個獨立 Module，所以請確保每個 Module 的「目標」是一句明確的 thesis。

### Example

```md
# 方法論與工具箱深度解析 (Methodology & Toolchain Extraction)
**Paper:** [論文名稱]

*   **技術痛點：** // 描述過去傳統方法的瓶頸是什麼。
*   **解決方案：** // 一句話總結這篇論文用什麼高通量/新穎技術解決了上述痛點。

---

### 2. 實驗技術與細節 (Wet-lab Techniques & Protocols)

*   **目標：** // 為什麼要做這一步？這句話會成為下游的 Thesis。

    *   *優勢：除以 DNA reads 可以標準化細胞數量的微小差異。* // 補充這樣計算的好處。

#### B. [統計模型，例如：統計與線性建模]
*   **目標：** // 這句話會成為下游的 Thesis。
*   **技術細節與參數：**
    *   **工具：** R 語言的 `lm` 函數。

---

    *   **Step 2:** ...
    *   **效果：** 解決了傳統隨機插入無法控制變數的致命傷。 // 重點：必須解釋組合起來「解決了什麼原本無法解決的問題」。

---

如果你未來的研究遇到以下狀況，可以借鑑這篇論文的方法：
```

## 規則

* **乾濕絕對分離**：Wet-lab (細胞、病毒、硬體) 與 Dry-lab (統計、R語言、演算法) 必須分置於第 2 節與第 3 節。不准把軟體套件跟細胞培養寫在同一個標題下。
* **強制溯源**：在第 2 節與第 3 節中，只要作者提到使用了某項既有技術，你必須在文中尋找文獻引用（如 `Smith et al.` 或 `Addgene ID`）並標註出來。
* **強制填寫參數**：不要只寫「作者使用了 PCR」或「作者做了 FACS」。必須寫出「用什麼 Enzyme 進行 PCR，溫度多少」或「FACS 篩選了什麼螢光標記，排除了什麼」。如果 Online Methods 有寫，就必須萃取。
* **Thesis 清晰度**：第 2 節與第 3 節各個子項的 **目標** 欄位會直接作為下游 Worker 的 `thesis`，請確保它是一句完整、紙本導向的敘述，例如「在細胞基因組中隨機且單一地插入帶有條碼的『插槽』，作為後續實驗的基礎。」
* **Heading 純中文**：第 2 / 3 節各子項標題（`#### A. ...`）寫成純中文（必要的英文專有名詞、套件名、酵素名可內嵌），**禁止**寫成「中文 (English)」或「English (中文)」對照式雙語格式。下游 merger 會把它顯示為「主題N：[heading]」，雙語對照會讓顯示混亂。
* **Pipeline 要寫出「Why」**：第 4 節不能只是把前面的步驟重抄一遍。必須解釋步驟之間為何如此搭配（例如：「利用 FACS 的負向篩選，完美解決了前一步驟 Cre-lox 置換效率過低的問題」）。
* **情境化 Takeaway**：第 5 節不要寫空泛的「這個技術很好」。必須寫出讀者未來可能遇到的「痛點情境」，以及這篇論文的思路如何套用在該情境上。

## 銜接：執行階段

本規劃輸出會交給 Orchestrator (`method_SKILL.md`)，由其解析第 2 節與第 3 節的每一個 A, B, C... 子項，把它們當成獨立模塊（Module）派發給 `method_worker` 進行詳細展開，並由 Reviewer 評審。
